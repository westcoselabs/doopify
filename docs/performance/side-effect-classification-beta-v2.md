# Doopify Beta/V2 Side-Effect Classification (Phase E1)

Date: 2026-05-19  
Branch context: `beta/v2`  
Scope: classification + worker-planning input only (no runtime changes)

## Classification Legend

1. `inline-critical`: must stay in request path for correctness/security.
2. `safe-to-queue`: can be deferred without changing payment/order/inventory truth.
3. `already-queued`: already persisted as background work before external side effect runs.
4. `worker-candidate`: good candidate for dedicated worker execution once hardening gates are met.
5. `needs-idempotency-hardening-before-worker`: do not move yet; hardening first.

## Key Findings

- Checkout/payment/order/inventory truth paths remain correctly separated from side-effect fan-out.
- Many non-critical side effects are already queued (`Job`, `OutboundWebhookDelivery`, `WebhookDelivery`), which is strong groundwork for future worker extraction.
- Two reliability gaps should be treated as pre-worker hardening gates:
  - `Job` rows can remain stuck in `RUNNING` if a runner crashes (no stale-lock reclaim).
  - retry claim paths can orphan claimed work if a process crashes mid-attempt (inbound and outbound retry loops).
